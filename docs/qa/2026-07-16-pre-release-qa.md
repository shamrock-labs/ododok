# 오도독 iOS 1.0.1 출시 전 QA

이 문서는 `origin/main`의 `76b2b41`을 App Store 출시 후보로 검증한 결과와 남은 수동 QA 시나리오를 정의한다.

## 독자와 판정 기준

- 독자: iOS 개발자, 백엔드 개발자, 출시 승인자, 실기기 QA 담당자
- 목적: 출시 차단 결함을 먼저 제거하고, 실기기·실계정·App Store Connect가 필요한 항목을 누락 없이 실행한다.
- P0: 개인정보 손실·삭제 실패, 인증 우회, 앱 사용 불가처럼 즉시 출시를 막는 문제
- P1: 핵심 흐름 실패 또는 회귀 탐지 체계 붕괴로 출시 전에 해결하거나 명시적으로 승인해야 하는 문제
- P2: 우회 가능한 기능·표시 문제로 수정 일정과 운영 대응을 정해야 하는 문제
- P3: 출시 후 정리 가능한 경고·유지보수 문제

## 발견한 문제

### QA-REL-001 계정 삭제가 서버 성공 전에 완료 처리된다 — iOS 수정 완료, 수동 회귀 필요

- 우선순위: P0
- 기존 증상: 사용자가 계정 삭제를 누르면 로컬 토큰과 상태가 즉시 지워지고 로그인 화면으로 이동했다. 서버 삭제는 detached task에서 한 번 실행되며 오류를 `try?`로 버렸다.
- 위험: 오프라인, 서버 5xx, 앱 강제 종료가 겹치면 사용자는 삭제가 끝난 것으로 보지만 서버 계정과 씹기 기록이 남을 수 있다. 재시도할 인증 토큰도 이미 로컬에서 제거된다.
- 수정: `AppState.eraseAllUserData()`가 `DELETE /v1/me`의 2xx 응답을 먼저 기다리고, 성공한 경우에만 로컬 상태와 토큰을 삭제한다. 실패 시 로그인 상태를 유지하고 설정 화면에 재시도 다이얼로그를 표시하며, 요청 중 중복 입력과 시트 닫기를 막는다.
- 자동 검증: 계정 삭제 관련 16개 테스트와 전체 유닛 테스트 344개가 통과했다. 서버 실패 시 토큰·로그인·프로필·도토리·스트릭 유지와 완료 Analytics 미전송을 검증했다.
- 실기기 검증: 정상 네트워크에서 계정 삭제와 동일 카카오 계정 재로그인이 통과했다.
- 남은 출시 조건: 실기기에서 오프라인, 5xx, 요청 중 앱 종료 UX를 수동 확인한다. 서버의 실제 보관 범위 검증은 QA-REL-002 및 SET-04에 남는다.
- 필수 회귀: 정상 삭제, 오프라인 삭제, 401 후 refresh 성공, refresh 실패, 5xx, 요청 중 앱 종료, 중복 탭 멱등성을 검증한다.

### QA-REL-002 원시 IMU 업로드와 개인정보 선언 범위가 일치하지 않는다

- 우선순위: P0
- 증상: 개인화 측정 산출물은 Application Support에 저장되고 서버 업로드 실패 시 다음 진입까지 보존된다. 산출물에는 원시·가공 IMU CSV와 진단 데이터가 포함될 수 있다.
- 위험: 앱 내 개인정보처리방침은 모션 신호를 “분석”한다고만 설명하고 서버 전송·보관·재시도 저장을 명확히 고지하지 않는다. `PrivacyInfo.xcprivacy`에도 센서·피트니스 데이터 수집 유형이 없다.
- 추가 불일치: 개인정보처리방침은 마이크 권한을 요청할 수 있다고 적지만 앱에는 `NSMicrophoneUsageDescription`이나 마이크 권한 요청 경로가 없다.
- 추가 검토: Analytics는 서버 사용자 ID를 `setUserId`로 지정하고 익명 device ID를 사용자 속성으로 전송한다. manifest의 Product Interaction과 Device ID가 `Linked=false`인 선언이 실제 처리 방식과 맞는지 확인해야 한다.
- 출시 조건: 실제 서버 저장 정책·보유 기간·삭제 범위를 확정하고 앱 내 방침, App Store 개인정보 라벨, privacy manifest를 같은 내용으로 맞춘다. 마이크를 쓰지 않으면 고지에서 제거한다.

### QA-REL-003 핵심 UI 테스트 8개가 제품 회귀를 판정하지 못한다

- 우선순위: P1
- 실패 목록: 홈 식사 시작/종료 2개, 푸시 시작 강조 2개, 기록 빈 상태 1개, 설정·계정 삭제 3개
- 원인 A: 홈·푸시 테스트는 `-resetState`로 로그아웃한 뒤 `-forceLogin`을 전달하지 않아 로그인 화면에서 버튼을 찾는다.
- 원인 B: 설정·기록 테스트는 `-skipOnboarding`을 설정하지만 비동기 `NoopAuthSessionManager.me()`가 `onboardingCompleted=false`를 반환해 상태를 다시 덮는다. 온보딩 sheet가 핵심 화면을 가린다.
- 추가 노후화: 계정 삭제 취소 테스트는 시스템 `confirmationDialog`의 `PopoverDismissRegion`을 기대하지만 현재 구현은 커스텀 `appDialog`다.
- CI 영향: `ios-ci.yml`은 유닛 테스트 스킴만 실행해 이 실패를 감지하지 않는다.
- 출시 조건: 테스트 launch state를 한 소유 위치에서 결정적으로 만들고 16/16을 통과시킨다. 이후 핵심 UI suite를 CI 필수 체크에 포함한다.

### QA-REL-004 기록 범위 조회 실패가 사용자에게 드러나지 않는다

- 우선순위: P2
- 증상: Report Hub의 범위 세션 조회가 `try?` 실패를 빈 배열로 바꾼다.
- 위험: 오프라인·서버 오류와 실제 기록 없음이 구분되지 않는다. 기존 캐시는 남을 수 있지만 새 범위 조회 실패를 사용자가 알 수 없다.
- 출시 조건: 조회 실패 상태와 재시도 동작을 제공하고, 마지막 정상 데이터와 오류 안내를 함께 유지한다.

### QA-REL-005 출시 검증 자동화에 lint와 UI suite가 없다

- 우선순위: P2
- 증상: 프로젝트 규칙은 SwiftLint 통과를 요구하지만 CI에 lint 단계가 없고 현재 로컬에도 도구가 없다. UI suite도 CI에서 제외돼 있다.
- 위험: 새 lint 경고, 접근성 identifier 회귀, 주요 화면 진입 실패가 main에서 통과할 수 있다.
- 출시 조건: 고정 버전 SwiftLint와 안정화된 UI smoke suite를 CI에 추가한다.

### QA-REL-006 Amplitude 초기화 API가 deprecated 상태다

- 우선순위: P3
- 증상: Release 빌드와 정적 분석에서 `Configuration.defaultTracking`과 이전 initializer 경고가 반복된다.
- 영향: 현재 빌드는 성공하지만 SDK 상향 시 컴파일 오류로 바뀔 수 있다.
- 조치: `AutocaptureOptions` 기반 API로 이전하고 세션·lifecycle 이벤트 중복 여부를 검증한다.
